# 报告AI解读系统 - 实施手册（实施人员）

面向现场实施人员：上线前配置检查、首次验证、LIS 对接配置、常见问题与联系支持。

## 〇、交付清单与上线前置条件

### 交付清单（部署完成后确认）

| 角色 | 完成项 |
|------|--------|
| **部署人员** | 镜像构建并推送；Secret（DATABASE_URL 等）与 ConfigMap 已创建；PostgreSQL 可用；Backend/Frontend 已部署；Ingress 或访问入口已配置。 |
| **实施人员** | 医院名称（HOSPITAL_NAME）、推理服务地址（VLLM_BASE_URL）已按本院填写；LIS/MSSQL 按需配置（见下文 1.3、1.7）。 |
| **业务验证** | 首页可打开；演示号（100001/100002/100003）或真实 HID 可搜到患者；能选择报告并成功获得 AI 解读；对接 LIS 时左侧可展示 PDF（在 LIS 返回 FILEURL 的前提下）。 |

### 上线前置条件（正式给用户使用须同时满足）

- **PostgreSQL**：生产必须使用 PostgreSQL，由部署人员配置；不得使用 SQLite。
- **推理服务可达**：**VLLM_BASE_URL** 必须指向院内已部署的安诊儿推理服务。**本次实施采用 sglang 部署安诊儿模型**（本系统按 OpenAI 兼容 chat 接口调用）。未配置或不可达时，AI 解读会失败，仅适合演示/联调。

## 一、上线前配置检查清单

部署完成后，与信息中心或部署人员确认以下配置已按本院环境填写。

### 1.1 必填项

| 配置项 | 说明 | 示例/位置 |
|--------|------|-----------|
| 医院名称 | 界面顶部及标题展示 | ConfigMap `HOSPITAL_NAME` 或 .env `HOSPITAL_NAME` |
| 推理服务地址 | 安诊儿模型地址（**本次实施为 sglang 部署**），解读依赖此项 | `VLLM_BASE_URL`，如 `http://10.x.x.x:8000` |
| 数据库 | 生产必须为 PostgreSQL，连接串由部署在 Secret 中配置 | 部署人员负责，实施只需确认“已用 PostgreSQL” |

### 1.2 选填项（按本院是否有）

| 配置项 | 说明 | 何时必填 |
|--------|------|----------|
| OCR 服务地址 | 上传报告图片、PDF 解析 | 需要“上传外院报告图片解读”或 LIS 返回 PDF 解析时 |
| LIS 对接 | 卫宁等 LIS 报告列表与 PDF | 需要从本院 LIS 拉取报告、左侧展示 PDF 时 |

### 1.3 LIS 对接（卫宁 ASMX）配置

若本院使用卫宁 LisWebService.asmx（入参 HID，返回报告列表及 FILEURL）：

| 配置项 | 值 | 说明 |
|--------|-----|------|
| LIS_ADAPTER | winning | 使用卫宁适配器 |
| LIS_USE_ASMX | true | 启用 ASMX 调用方式 |
| LIS_API_BASE_URL | 卫宁 ASMX 地址 | 如 `http://10.8.80.23:8092/LisWebService.asmx` |
| LIS_ASMX_METHOD_NAME | 方法名 | 通常为 `GetReportList`，以卫宁文档为准 |
| LIS_ASMX_JSON_PARAM_NAME | 入参名 | 通常为 `jsonParam`，入参为 `{"HID":"挂号/首页序号"}` |

- 以上在 **K8s** 中为 ConfigMap（及必要时 Secret 中的 LIS_API_KEY）；**Docker Compose** 中为环境变量或 .env。
- 实施只需确认信息中心已提供正确的“调用地址、方法名、入参名”，并由部署人员写入配置。

### 1.4 嵌入/患者接口展示模式（按医院可配置）

对接患者接口或 EMR 弹窗时，弹出界面可配置为两种方式之一：

| 配置项 | 值 | 说明 |
|--------|-----|------|
| EMBED_REPORT_MODE | list_select | **先展示报告列表，选中哪条解读哪条**（推荐对接患者接口时使用） |
| EMBED_REPORT_MODE | latest_auto | 弹出后自动解读该患者**最新一条**报告 |

- 配置方式：环境变量或 K8s ConfigMap 中设置 `EMBED_REPORT_MODE=list_select` 或 `latest_auto`。
- **患者接口直接打开**：若 EMR/工作站通过链接打开解读界面，可使用固定“列表选读”的地址，不受上述配置影响：  
  `https://系统地址/patient/住院号或门诊号`  
  例如 `https://报告AI解读域名/patient/4435923`，打开即展示该患者报告列表，点击某条即解读该条。

### 1.5 全站中文与时区

- **界面与接口**：全站中文展示；API 响应统一 **UTF-8**，日期时间按 **中国时区（Asia/Shanghai）** 序列化与展示，避免乱码和时区混乱。
- **后端**：配置项 `TIMEZONE=Asia/Shanghai`（默认已设），可在 .env 或 ConfigMap 中覆盖。
- **前端**：日期使用 `zh-CN` 与 `Asia/Shanghai` 格式化；请求头带 `Accept-Language: zh-CN`。
- 若控制台/日志出现中文乱码，可设置环境变量 `PYTHONIOENCODING=utf-8`（或 Windows 下执行 `chcp 65001`）后再启动后端。

### 1.6 轻量知识增强（规范与指南）

若希望解读时**有据可依**，可启用「轻量知识增强」：在 `knowledge/` 目录下按报告类型、科室放置规范/指南摘录（Markdown），系统会在生成解读时自动注入到模型上下文中。

| 配置项 | 说明 | 示例 |
|--------|------|------|
| KNOWLEDGE_ENABLED | 是否启用知识注入 | true |
| KNOWLEDGE_DIR | 知识目录（相对后端项目根） | knowledge |
| KNOWLEDGE_MAX_CHARS | 注入内容总长度上限（字符） | 4000 |

- **目录约定**：`knowledge/report_type/{报告类型}/*.md`、`knowledge/department/{科室}/*.md`。报告类型与科室编码与系统一致（如 lab、hematology）。详见后端 `knowledge/README.md`。
- **内容建议**：摘录临床指南、操作规程、危急值标准等关键段落；正式使用前请替换示例文件为院内认可规范。

### 1.7 SQL Server：病历号/门诊卡号 → HID 解析（正式对接）

当 EMR 传的是**病历号/门诊卡号（pat_num）**而 LIS（卫宁）需要**首页序号/挂号序号（xh，即 HID）**时，可启用本系统内置的 **SQL Server 解析**：根据 pat_num 查视图得到 xh，再用 xh 调卫宁 LIS 获取报告。

| 配置项 | 说明 | 示例 |
|--------|------|------|
| MSSQL_ENABLED | 是否启用 pat_num→xh 解析 | true |
| MSSQL_SERVER | SQL Server 服务器地址 | 10.8.80.21 |
| MSSQL_USER | 数据库用户名 | AISET |
| MSSQL_PASSWORD | 数据库密码（务必放入 .env 或 K8s Secret，勿提交代码库） | 由院方提供 |
| MSSQL_DATABASE | 数据库名 | THIS4 |
| MSSQL_VIEW_HID | 视图名（存 xh 与 pat_num 对应关系） | VW_BRJZXXK |
| MSSQL_COLUMN_PAT_NUM | 视图中医案号/门诊卡号列名 | pat_num |
| MSSQL_COLUMN_XH | 视图中首页序号/挂号序号列名 | xh |

- **逻辑**：API 收到 `patient_id`（或搜索 keyword）时，若启用 MSSQL，先执行 `SELECT TOP 1 xh FROM VW_BRJZXXK WHERE pat_num = ?`，得到 xh 后再用 xh 调 LIS；未启用或未查到则把传入值当作 HID 直接查 LIS。
- **部署**：K8s 下将 `MSSQL_PASSWORD` 等敏感项放入 Secret，其余可放 ConfigMap。后端需能访问 SQL Server 网络（防火墙/安全组放行）。

**LIS 与 MSSQL 适用关系**：若 EMR 传的是**病历号/门诊卡号（pat_num）**而 LIS（卫宁）需要**首页序号/挂号序号（HID）**，需同时配置 **LIS（卫宁）** 与 **MSSQL（pat_num→xh 解析）**；若 EMR 直接传 HID，则仅配置 LIS 即可。

**启用 MSSQL 时构建或运行报错**：后端使用 pymssql 连接 SQL Server。若 Docker 镜像构建或运行时报相关错误，可能需在 Dockerfile 或基础镜像中安装 **freetds** 等系统依赖（如 Debian/Ubuntu：`apt-get install -y freetds-dev`），再安装 pymssql。具体以报错信息为准，必要时联系部署/开发。

## 二、首次使用验证

### 2.1 打开系统

- 浏览器访问部署提供的地址（如 `https://report-ai.医院域名` 或 `http://服务器IP:80`）。
- 应看到“报告AI解读”相关标题及首页（患者搜索框、“上传外院报告图片解读”等）。

### 2.2 演示模式（未对接 LIS）

- 在搜索框输入：`100001`、`100002` 或 `100003`，点击搜索。
- 选择患者进入解读页，选择报告后点击“开始AI解读”。
- 若右侧出现“异常总结”“临床意义”“临床建议”等，说明推理服务与基本流程正常。

### 2.3 已对接 LIS（卫宁 ASMX）

- 使用本院真实**住院首页序号**或**门诊挂号序号**（即 HID）在首页搜索。
- 应出现该患者的报告列表；点击某条报告后，左侧应能展示 **PDF 报告**（若卫宁返回了 FILEURL），右侧为 AI 解读结果。
- 若报告列表为空，请核对 HID 是否正确、LIS_API_BASE_URL 与 ASMX 方法名是否与卫宁文档一致。

### 2.4 健康检查（可选）

- 访问：`/api/v1/system/health`（与系统同域名，路径前加系统根地址）。
- 可查看 `vllm_status`、`ocr_status`、`lis_adapter` 等，便于判断推理、OCR、LIS 是否可用。

## 三、模拟数据验证（开发/联调）

用于在**未对接真实 LIS** 时，用内置 Mock 数据跑通全流程，验证产品完整性、健壮性。

### 3.1 前置条件

- 后端使用 **Mock LIS**：`LIS_ADAPTER=mock`（.env 或环境变量）。
- 数据库：本地可用默认 SQLite（`DATABASE_URL=sqlite+aiosqlite:///./report_ai.db`）。
- **AI 解读**：若未部署 sglang（安诊儿），解读接口会返回 500，脚本中该项按「跳过」处理，其余接口仍可全部校验。

### 3.2 Mock 数据说明

| 患者 ID | 姓名 | 报告编号 | 报告名称 |
|---------|------|----------|----------|
| 100001 | 张* | RPT20260301001 | 生化全套 |
| 100002 | 李* | RPT20260305001 | 血常规 |
| 100003 | 王* | RPT20260308001 | 血气分析+感染标志物 |

- Mock 适配器**不提供 PDF**，故「报告 PDF 代理」接口预期返回 501，属正常。

### 3.3 自动校验脚本（推荐）

1. 启动后端（在项目根目录或 backend 目录）：
   ```bash
   cd backend && python -m uvicorn app.main:app --host 0.0.0.0 --port 8080
   ```
2. 另开终端，在项目根目录执行：
   ```bash
   python scripts/verify_simulation.py --base-url http://127.0.0.1:8080
   ```
3. **预期结果**：共 7 项检查，全部通过或仅「AI 解读」因无 sglang（安诊儿）跳过。若出现 0 失败即表示 API 层完整、健壮性符合预期。

脚本覆盖：健康检查、前端配置、患者搜索、报告列表、不存在患者 404、AI 解读（有 sglang/安诊儿 则通过否则跳过）、PDF 代理在 Mock 下 501。

### 3.4 前端手工验证

1. 启动前端：`cd frontend && npm run dev`，浏览器访问前端开发地址（本仓库默认 `http://localhost:3000`，见 `frontend/vite.config.js`）。
2. 前端通过 Vite 代理将 `/api` 转发到后端 `http://localhost:8080`，无需额外配置。
3. **首页**：搜索 `100001` → 应出现「张*」，进入后应看到报告「生化全套」，点击「开始AI解读」→ 有 sglang（安诊儿）时右侧出现解读，无则提示错误。
4. **患者接口**：直接访问 `http://localhost:3000/patient/100001` → 应直接展示该患者报告列表，点击某条再解读。
5. **嵌入页**：访问 `http://localhost:3000/embed-inner?pid=100002` → 应展示李*的报告列表，选中后可解读。

通过以上步骤可确认：列表拉取、患者解析、解读链路、患者/嵌入路由与展示模式均正常。

## 四、常见问题

| 现象 | 可能原因 | 建议 |
|------|----------|------|
| 首页打不开 | 地址错误、Ingress/网关未生效 | 确认访问地址与部署文档一致；联系部署人员检查 Ingress/防火墙。 |
| 搜索患者无结果 | Mock 下未输 100001/100002/100003；对接 LIS 时 HID 错误或 LIS 未配置 | Mock 用演示号；对接 LIS 时用真实 HID 并确认 LIS 配置正确。 |
| 报告列表为空（已对接 LIS） | HID 不对、ASMX 地址/方法名错误、网络不通 | 与信息中心确认 HID 含义及 ASMX 调用方式；让部署检查 LIS_API_BASE_URL、LIS_USE_ASMX、方法名。 |
| 左侧没有 PDF | 该报告无 FILEURL 或 LIS 未返回 FILEURL | 确认卫宁接口返回中有 FILEURL；若仅部分报告有，属正常。 |
| “开始AI解读”一直转圈或报错 | 推理服务不可用或超时 | 检查 VLLM_BASE_URL 是否可达；查看后端日志；确认 sglang（安诊儿）服务正常。 |
| 上传图片解读失败 | OCR 未配置或不可达 | 配置 OCR_SERVICE_URL 并确认服务可用；或暂时不使用“上传图片解读”功能。 |

## 五、配置修改后是否需要重启

- **K8s**：修改 ConfigMap 或 Secret 后，需重启后端 Pod 使配置生效，例如：  
  `kubectl -n report-ai rollout restart deployment report-ai-backend`
- **Docker Compose**：修改 .env 或环境变量后，需执行 `docker-compose up -d` 重新创建后端容器。
